📊 데이터 전처리 및 통계 검정 관련
Q1. 카이제곱 검정 시, 모든 셀의 기대빈도가 5 이상이어야 한다는 전제조건을 확인했나요?
답변: 네, 전수 조사하여 확인했습니다.

SciPy 라이브러리의 stats.chi2_contingency() 함수가 반환하는 expected(기대빈도 행렬) 변수를 직접 확인했으며, 심근경색 유병률이 낮음에도 불구하고 모든 셀의 기대빈도가 5를 확실히 초과함을 검증했습니다.

Q2. p-value가 지나치게 잘 나왔는데, 전처리 과정에서 데이터 누수(Data Leakage)는 없었나요?
답변: 데이터 누수가 아니라, 철저한 문헌 고찰 기반의 변수 설계 덕분입니다.

변수 선정 단계에서 심근경색 유병률 관련 기존 의학 논문들을 정밀 분석했습니다. 임상적 연관성이 이미 입증된 핵심 지표들만 엄선하여 모델에 반영했기 때문에 통계적으로 유의미한 결과(낮은 p-value)가 나온 것입니다.

Q3. 단순 SMOTE를 쓰지 않고, 다수 클래스(0)를 10,000개로 줄인 후 SMOTE를 적용한 이유는 무엇인가요?
답변: 과적합(Overfitting)과 가상 데이터의 노이즈를 방지하기 위해서입니다.

원본 국민건강영양조사 데이터는 정상군(0)의 비율이 압도적으로 높은 극심한 불균형 상태입니다. 이 상태에서 바로 SMOTE를 적용하면 소수 클래스(1)의 가상 데이터를 지나치게 많이 생성하게 되어 노이즈가 심해지고 과적합 위험이 커집니다. 따라서 다수 클래스를 10,000개로 1차 다운샘플링한 후 균형을 맞추었습니다.

Q4. 범주형 변수에 원-핫 인코딩(One-Hot Encoding)을 적용한 이유는 무엇인가요?
답변: 범주형 데이터에 모델이 임의의 서열이나 가중치를 부여하는 것을 막기 위해서입니다.

예를 들어 '교육수준'이 1(초졸), 2(중졸), 3(고졸)로 되어 있을 때, 이를 그대로 사용하면 모델이 초졸(1) + 중졸(2) = 고졸(3)과 같은 잘못된 산술 연산을 할 수 있습니다. 이를 방지하고 각 범주를 독립된 특성으로 다루기 위해 원-핫 인코딩을 수행했습니다.

Q5. 모델 훈련 시 데이터 정규화(Normalization/Scaling)를 하지 않은 이유는 무엇인가요?
답변: 선택한 모델들이 모두 '트리 기반 모델'이기 때문입니다.

트리 기반 모델(XGBoost, LightGBM, Random Forest)은 거리나 수식을 연산하는 방식이 아니라, 특정 임계값을 기준으로 대소 관계를 비교하며 데이터를 분기(Split)합니다. 따라서 데이터의 스케일(단위)이 피처 중요도나 분기 기준에 영향을 미치지 않으므로 정규화가 불필요합니다.

🤖 모델링 및 알고리즘 관련
Q6. LightGBM, XGBoost, Random Forest 3가지 모델을 선정한 이유는 무엇인가요?
답변: 각 모델의 강점을 조합하여 불균형 의학 데이터 예측을 최적화하기 위해서입니다.

LightGBM: 데이터 불균형 상황에서 살아난 소수 클래스(심근경색 발병군)의 미세한 특성을 빠르게 포착합니다.

XGBoost: 의학 데이터 특성상 발생하는 변수 간의 복잡한 상호작용을 정교한 트리 구조로 잡아내는 능력이 탁월합니다.

Random Forest: 부스팅 모델(XGBoost, LightGBM)들이 증강된 데이터에 과적합(Overfitting)되고 있는지 판단하는 안정적인 기준점(Baseline)으로 활용했습니다.

Q7. XGBoost를 최종 모델로 선택한 구체적인 근거는 무엇인가요?
답변: 분류 성능을 나타내는 ROC-AUC 지표가 가장 우수했기 때문입니다.

초기 모델 비교 시 XGBoost가 ROC-AUC 0.9326으로 LightGBM(0.9003) 및 Random Forest(0.9046)보다 높은 성능을 보였습니다. 이후 하이퍼파라미터 튜닝(GridSearchCV)을 거쳐 최종 ROC-AUC 0.9801, 정확도 0.90을 달성하여 최종 모델로 채택했습니다.

Q8. 하이퍼파라미터 중 eval_metric = "logloss"의 역할은 무엇인가요?
답변: 모델이 예측한 확률과 실제 정답 간의 오차를 촘촘하게 계산하여 학습하도록 돕는 지표입니다.

잘못 예측한 데이터에 대해 더 큰 페널티를 부여함으로써, 모델이 손실(Loss)을 최소화하는 방향으로 다음 트리를 더 정교하게 생성하도록 유도합니다.

Q9. 사용한 하이퍼파라미터들의 의미를 간략하게 설명해 주세요.
답변: * n_estimators: 모델에 생성할 총 결정 트리(나무)의 개수

learning_rate: 이전 트리의 오차를 다음 트리에 반영할 때 적용하는 학습률(가중치 크기)

max_depth: 각 트리가 가질 수 있는 최대 깊이

min_child_weight: 자식 노드가 분기하기 위해 필요한 최소 가중치(샘플 수)의 합 (과적합 방지용)

subsample: 각 트리를 학습시킬 때 사용할 무작위 데이터 행(샘플)의 비율

colsample_bytree: 각 트리를 학습시킬 때 사용할 무작위 변수 열(피처)의 비율

Q10. 평가지표 중 macro avg와 weighted avg 차이는 무엇인가요?
답변: * macro avg: 클래스별 데이터 개수를 고려하지 않고, 각 클래스의 지표를 단순 평균 낸 값입니다. (소수 클래스의 성능이 잘 반영됨)

weighted avg: 각 클래스가 가진 데이터 개수(가중치)를 고려하여 평균을 낸 값입니다. (다수 클래스의 성능에 영향을 많이 받음)

Q11. XGBoost의 작동 방식과 한계(블랙박스)에 대해 설명해 주세요.
답변: 여러 개의 결정 트리를 순차적으로 학습시키는 앙상블(부스팅) 기법입니다.

이전 트리가 틀린 오차를 다음 트리가 집중적으로 보완하는 방식으로 작동합니다. 수학적 매커니즘은 명확하여 예측 성능이 매우 높지만, 수백 개의 트리가 복잡하게 얽혀 있어 "왜 이 환자를 심근경색으로 예측했는가?"에 대한 인과관계를 사람이 직접 추적하기는 어렵습니다. 이를 블랙박스 모델이라고 하며, 설명 가능성(Explainability)이 낮다는 한계가 있습니다.

⚠️ 심층 압박 면접 대비 (한계점 및 논리 방어)
Q12. 심근경색 클래스(1)의 Precision(정밀도)이 0.34~0.39로 낮은데, 오탐률로 인한 사용자 신뢰도 저하 문제는 없나요?
답변: 본 서비스는 '의학적 진단'이 아닌 '행동 변화를 유도하는 보조 지표'가 목적이므로, Precision보다 Recall(재현율)이 더 중요합니다.

실제 환자를 놓치는 위험(진양성을 놓침)을 줄이기 위해 재현율을 확보하는 과정에서 정밀도가 낮아졌습니다. 위험군 오분류로 인한 불편함보다는, 잠재적 위험을 조기에 인지하고 병원 방문 및 정기 검진을 유도하는 공익적 가치와 참고 자료로서의 역할에 집중하고 있습니다.

Q13. 실제 유병률은 5.7%인데 SMOTE로 1:1 비율을 맞추면 현실을 왜곡하는 것 아닌가요?
답변: SMOTE는 어디까지나 '모델 학습용' 데이터 증강 기법입니다.

극심한 데이터 불균형 상태에서는 모델이 다수 클래스(정상)로만 찍어도 정확도가 높게 나오는 오류가 발생합니다. 학습 단계에서는 1:1로 균형을 맞춰 양성 클래스의 특징을 명확히 배우게 하고, 최종 모델 검증(Test Set) 단계에서는 원본 데이터 비율을 그대로 유지하여 평가했기 때문에 현실 검증력을 갖추고 있습니다.

Q14. GridSearchCV 수행 시 과적합(Overfitting) 여부는 어떻게 검증했나요?
답변: 3-Fold 교차 검증(cv=3)을 수행하였고, Train/Val/Test 셋의 ROC-AUC 스코어를 비교하여 검증했습니다.

검증 결과는 아래와 같습니다.

Train ROC-AUC: 0.9983 / Val ROC-AUC: 0.9801

Train과 Val의 차이가 0.0182(0.05 이하)로 내부 과적합이 없음을 확인했습니다.

최종 Test ROC-AUC 역시 0.9238로 준수한 일반화 성능을 보였습니다.

Q15. 공복혈당 이상치(Outlier)를 제거하지 않고 유지한 근거와 모델에 미친 영향은?
답변: 임상적으로 '초고위험군 환자의 실제 데이터'일 가능성이 높기 때문입니다.

박스플롯 분석 결과 수치가 극단적으로 높게 나왔지만, 이는 데이터 오류가 아니라 심한 당뇨나 고혈당을 앓고 있는 실제 고위험군의 데이터로 해석했습니다. 심근경색과 고혈당은 밀접한 연관이 있으므로, 이 이상치를 제거하면 오히려 고위험군을 예측하는 중요한 정보(시그널)가 손실되어 모델의 예측력이 떨어질 수 있어 유지했습니다.

Q16. 변수 중요도에서 고혈압, 당뇨보다 이상지질혈증(DI2_dg), 흡연, 연령이 더 높게 나온 의학적 이유는 무엇인가요?
답변: 변수 중요도는 의학적 절대 기준이 아니라, '해당 데이터셋 내에서의 통계적 기여도'를 의미합니다.

고혈압과 당뇨 역시 유의미한 변수로 학습되었으나, 저희가 사용한 국민건강영양조사 데이터 내에서는 이상지질혈증 여부, 흡연 여부, 연령의 분포가 심근경색 발생 여부와 훨씬 더 강력한 통계적 상관관계를 보였기 때문에 모델이 이를 주요 분기 기준으로 선택한 것으로 해석됩니다.

Q17. 카이제곱/Mann-Whitney U test로 변수를 골랐는데, 공복혈당-당화혈색소 같은 다중공선성 문제는 확인했나요?
답변: 트리 기반 모델을 사용했기 때문에 다중공선성 문제로부터 자유롭습니다.

다중공선성은 독립변수 간 회귀계수의 독립성을 가정하는 '선형 모델'에서 주로 문제가 됩니다. XGBoost와 같은 트리 기반 모델은 변수를 순차적으로 선택하며 분기하기 때문에, 변수 간 상관관계가 높더라도 모델의 예측 성능이나 알고리즘 작동에 직접적인 악영향을 주지 않습니다.

Q18. 2005~2013년 데이터를 심근경색 진단자 중심으로 추출하면 양성 편향(Bias)이 생기지 않나요?
답변: 소수 클래스 보강을 위한 불가피한 선택이었으며, 데이터적 한계점으로 인지하고 있습니다.

초기 원본 데이터 내에 심근경색 환자 샘플이 너무 적어, SMOTE를 적용하기 전 최소한의 실제 양성 데이터를 확보하기 위해 해당 기간의 진단자 데이터를 집중 추출했습니다. 이로 인해 초기 데이터에 양성 편향이 발생할 수 있다는 점은 알고리즘의 한계점으로 인정하며, 향후 더 넓은 범위의 균형 있는 데이터 확보를 통해 보완해야 할 과제입니다.

Q19. 원-핫 인코딩된 변수들(사용자 입력 19개 ➡️ 내부 변수 35개)은 웹 아키텍처에서 어떻게 처리했나요?
답변: UI 편의성을 위해 백엔드와 AI 서버(Flask)의 역할을 분리했습니다.

사용자는 웹 프론트엔드에서 직관적인 19개의 문항에만 답변합니다. 이 데이터는 백엔드 Spring Boot를 거쳐 DB에 원본 그대로 저장됩니다. 이후 AI 연산을 담당하는 Flask 서버로 데이터가 넘어오면, Flask 내부 파이프라인에서 이를 35개의 원-핫 인코딩 벡터로 변환하여 학습된 모델에 투입하고 확률을 계산합니다.

Q20. 하루에 한 번만 예측이 가능하도록 제한한 이유는 무엇인가요? 사용자가 불편하지 않을까요?
답변: 민감한 건강 예측 서비스의 신뢰도를 지키고 무분별한 서버 자원 낭비를 막기 위함입니다.

사용자가 수치를 무작위로 바꾸며 반복 테스트할 경우 결과에 대한 불신이 생길 수 있습니다. 다만, 단순 오입력으로 인한 불편함을 해소하기 위해 기존 입력 데이터를 삭제하고 즉시 재예측할 수 있는 '삭제 후 재시도' 기능을 구현하여 사용성을 보완했습니다.

Q21. Google OAuth 로그인 외에, 혈액검사 등 민감한 건강 정보의 보안은 어떻게 처리했나요?
답변: 인증은 Google OAuth에 위임했으며, 향후 인프라 보안 강화를 계획 중입니다.

보안 신뢰성을 높이기 위해 검증된 Google OAuth로 로그인 보안을 처리했습니다. 현재 프로젝트 단계에서는 DB 접속 정보나 민감 키 관리 측면에서 보안적 한계가 있음을 인지하고 있습니다. 실제 서비스화를 진행한다면 AWS Secrets Manager나 환경변수(.env) 분리, 데이터 암호화(AES-256) 등 추가적인 보안 플러그인을 적용할 예정입니다.

Q22. 예측 결과가 단순 확률(%)로만 제공되면 사용자가 해석하기 어렵지 않나요? (예: 12%의 위험성 체감)
답변: 국민 평균 유병률 데이터와 시각적 그래프를 함께 제공하여 직관성을 높였습니다.

단순 "12%"라는 숫자만 보여주지 않고, 옆에 위험도 게이지 그래프를 배치했습니다. 이와 함께 국민건강영양조사 기준 국민 평균 유병률(5.7%)을 가이드라인으로 함께 제시하여, 사용자가 자신의 위험도가 평균 대비 어느 정도 수준(예: 평균의 약 2배)인지 직관적으로 체감할 수 있도록 UI를 구성했습니다.

Q23. 과거 데이터(2005~2024년) 중심이라 최신 생활습관 트렌드가 반영되지 않을 수 있는데, 시간적 대표성 문제는 어떻게 보완할 건가요?
답변: 지속적인 데이터 수집 및 주기적인 모델 재학습 파이프라인(MLOps) 구축으로 보완하겠습니다.

서비스 운영을 통해 실제 이용자들의 최신 건강 검진 데이터 및 생활습관 데이터가 축적될 것입니다. 이 누적 데이터를 기반으로 모델을 주기적으로 업데이트(Finetuning)함으로써, 변화하는 현대인의 식습관과 생활 패턴 변화 추세를 모델에 지속적으로 반영할 계획입니다.
